1. O que esse arquivo faz?

Esse arquivo cria a infraestrutura completa para rodar o serviço de produtos na AWS, usando CDK (Cloud Development Kit). Em vez de clicar manualmente no console da AWS, você escreve código TypeScript que monta tudo automaticamente.

Conceito: AWS CDK CDK (Cloud Development Kit) é uma ferramenta da AWS que permite criar recursos na nuvem usando código TypeScript/Python/Java etc. Você escreve a infraestrutura como código e o CDK converte em CloudFormation (JSON/YAML) que a AWS executa.

2. Os Imports

Cada import é um "módulo" da AWS que você vai usar para criar recursos:

ImportO que faz
ec2Rede: VPC, Security Groups, regras de firewall
ecrRepositório de imagens Docker (Elastic Container Registry)
ecsServiço que roda containers (Elastic Container Service)
logsLogs da aplicação no CloudWatch
elbv2Load Balancers (ALB e NLB) para distribuir tráfego

3. A Interface de Props

Define o que essa Stack precisa receber de outras Stacks para funcionar. É como a lista de ingredientes de uma receita: 

interface ProductsServiceStackProps extends cdk.StackProps {
    vpc:        ec2.Vpc;                           // A rede onde tudo vai rodar
    cluster:    ecs.Cluster;                      // Agrupador dos containers ECS
    nlb:        elbv2.NetworkLoadBalancer;        // Balanceador de rede (TCP)
    alb:        elbv2.ApplicationLoadBalancer;   // Balanceador HTTP
    repository: ecr.Repository;                  // Imagem Docker da aplicação
}
Por que receber da outra Stack e não criar aqui? VPC, Cluster, ALB e NLB são recursos compartilhados por vários serviços. Por isso ficam em outra Stack e são passados como parâmetro — evitando criar duplicatas e facilitando o reuso.

5. TaskDefinition — "O molde do container"

Define quanto recurso o container vai ter. No Fargate você não gerencia servidores — só informa CPU e memória.

const taskDefinition = new ecs.FargateTaskDefinition(this, "TaskDefinition", {
    memoryLimitMiB: 1024,          // 1 GB de RAM
    cpu: 512,                       // meio vCPU
    family: "products-service",     // nome identificador na AWS
})
PropriedadeValorO que significa
memoryLimitMiB10241 GB de memória RAM para o container
cpu5120,5 vCPU (1024 = 1 vCPU inteiro)
familyproducts-serviceNome que agrupa versões da task na AWS

6. LogDriver — "Para onde vão os logs"

Configura onde os logs da aplicação vão aparecer no CloudWatch Logs.

const logDriver = ecs.LogDrivers.awsLogs({
    streamPrefix: "ProductsService",
    logGroup: new logs.LogGroup(this, "LogGroup", {
        logGroupName:    "ProductsService",
        removalPolicy:   cdk.RemovalPolicy.DESTROY,        // apaga com a stack
        retention:       logs.RetentionDays.ONE_WEEK,       // mantém 7 dias
    })
});
Dica: RemovalPolicy.DESTROY Quando você rodar cdk destroy, o grupo de logs e todos os logs dentro dele serão deletados automaticamente. Útil para ambientes de estudo. Em produção, use RETAIN para não perder histórico de logs acidentalmente.

7. addContainer — "O container em si"

Conecta a imagem Docker (do ECR) à TaskDefinition e expõe a porta da aplicação:

taskDefinition.addContainer("ProductsService", {
    image: ecs.ContainerImage.fromEcrRepository(props.repository, "1.0.0"),
    containerName: "ProductsService",
    logging: logDriver,
    portMappings: [{
        containerPort: 8080,
        protocol:      ecs.Protocol.TCP,
    }]
})

Importante A versão "1.0.0" precisa existir no ECR. Sempre que quiser fazer deploy de uma nova versão, mude essa tag aqui ou use uma estratégia de tag dinâmica (ex: latest ou a hash do commit).

8. FargateService — "O serviço rodando"

Usa a TaskDefinition e mantém 2 cópias do container rodando em paralelo para alta disponibilidade: 

const service = new ecs.FargateService(this, "ProductsService", {
    serviceName:    "ProductsService",
    cluster:        props.cluster,
    taskDefinition: taskDefinition,
    desiredCount:   2,             // 2 instâncias em paralelo
})
Por que desiredCount: 2? Se um container cair, o outro continua atendendo enquanto o ECS sobe um novo automaticamente. Isso é chamado de Alta Disponibilidade.

9. Permissão + Security Group

props.repository.grantPull(taskDefinition.taskRole);

service.connections.securityGroups[0]
    .addIngressRule(ec2.Peer.ipv4(props.vpc.vpcCidrBlock), ec2.Port.tcp(8080))
LinhaO que faz
grantPullDá permissão IAM ao container de baixar a imagem do ECR
addIngressRuleAbre a porta 8080 apenas para IPs dentro da VPC (segurança)
Segurança importante A regra ipv4(props.vpc.vpcCidrBlock) significa que só recursos dentro da mesma VPC podem acessar o container diretamente. O tráfego externo só chega via Load Balancer.

10. ALB — Application Load Balancer

O ALB distribui requisições HTTP entre as instâncias do container. Funciona na camada 7 (entende HTTP, headers, cookies etc). 

// 1. Criar o listener (ponto de entrada)
const albListener = props.alb.addListener("ProductsServiceAlbListener", {
    port:     8080,
    protocol: elbv2.ApplicationProtocol.HTTP,
    open:     true,
})

// 2. Registrar os containers como alvos + configurar health check
albListener.addTargets("ProductsServiceAlbTarget", {
    targetGroupName:     "productsServiceAlb",
    port:                8080,
    targets:             [service],
    protocol:            elbv2.ApplicationProtocol.HTTP,
    deregistrationDelay: cdk.Duration.seconds(30),
    healthCheck: {
        interval:         cdk.Duration.seconds(30),
        enabled:          true,
        port:             "8080",
        timeout:          cdk.Duration.seconds(10),
        path:             "/health",
        healthyHttpCodes: "200",
    }
})
ConfiguraçãoValorO que faz
path/healthEndpoint que o ALB bate para checar se o container está vivo
interval30sFrequência das verificações de saúde
timeout10sTempo máximo de espera por resposta
healthyHttpCodes200Só considera saudável se retornar HTTP 200
deregistrationDelay30sAguarda conexões ativas finalizarem antes de remover instância

11. NLB — Network Load Balancer

O NLB opera na camada 4 (TCP). É usado quando outros serviços AWS (como API Gateway com VPC Link) precisam acessar o ALB de forma privada: 

const nlbListener = props.nlb.addListener("ProductsServiceNlbListener", {
    port:     8080,
    protocol: elbv2.Protocol.TCP,
})

nlbListener.addTargets("ProductsServiceNlbTarget", {
    port:            8080,
    targetGroupName: "productsServiceNlb",
    protocol:        elbv2.Protocol.TCP,
    targets: [
        service.loadBalancerTarget({
            containerName: "ProductsService",
            containerPort: 8080,
            protocol:      ecs.Protocol.TCP,
        })
    ],
})
ALB vs NLB — Qual a diferença?
ALB entende HTTP, aplica regras de roteamento por path/header, ideal para microsserviços web.
NLB é mais rápido e simples, trabalha só com TCP/UDP — usado como ponto de entrada para o API Gateway (VPC Link) que não consegue falar diretamente com ALBs privados.

  2. Defina a interface de props com os recursos compartilhados (vpc, cluster, alb, nlb, repository)
  3. Crie a FargateTaskDefinition com CPU e memória
  4. Configure o LogDriver apontando para um LogGroup
  5. Adicione o container com addContainer (imagem, porta, logs)
  6. Crie o FargateService com desiredCount: 2
  7. Dê permissão de pull no ECR e abra a porta no Security Group
  8. Configure o ALB listener e o target group com health check em /health
  9. Configure o NLB listener e target group apontando para o container via loadBalancerTarget
